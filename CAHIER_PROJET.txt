Projet : CRM SUDICONE

Résumé
-------
CRM SUDICONE est une application web légère (Node.js + Express + SQLite) conçue pour gérer les contacts, le pipeline de ventes (opportunités), la facturation (devis/factures), les tâches, workflows internes, services/packs vendus et les statistiques analytiques. L'interface front-end est statique (HTML/CSS/JS) servie par Express, l'API REST backend est organisée par routes (auth, contacts, opportunites, factures, taches, users, services, workflows, stats, export). La base de données est SQLite via better-sqlite3.

Objectifs du cahier
-------------------
- Documenter l'architecture actuelle
- Décrire le schéma de données et les entités principales
- Lister les routes API et leur responsabilité
- Exposer les fonctionnalités existantes et le flux utilisateur
- Proposer les fonctionnalités manquantes ou à améliorer (statistiques avancées, automatisation, sécurité, backup, etc.)
- Fournir un roadmap et priorisation
- Proposer 10 thèmes de licence basés sur ce projet, avec justification pour chacun

1. Architecture générale
------------------------
- Serveur : Node.js + Express
- Base : SQLite (fichier `data/crm.db`), pilote `better-sqlite3` (synchrone)
- Auth : sessions Express (`express-session`) stockées en mémoire (MemoryStore par défaut)
- Templates front : fichiers statiques HTML/CSS/JS dans `public/` (login.html + pages/*.html)
- Routes API : `server/routes/*.js` (RESTful)
- Initialisation DB : `server/db/init.js` lit `schema.sql` et exécute `seed.js` pour données tests
- Uploads : dossier `uploads/` pour pièces jointes
- Logs : console; pas de logger structuré actuellement

2. Schéma de données (entités principales)
-----------------------------------------
- users (id, nom, email, password_hash, role, actif, created_at)
- contacts (id, type, nom, entreprise, email, telephone, adresse, ville, pays, siret, secteur, site_web, code_client, responsable, statut_client, owner_id, created_at, updated_at)
- services (id, code, titre, categorie, description, prix_ht, duree_mois, actif)
- service_packs, pack_services
- contact_services, contact_packs
- departments (org chart)
- workflows, workflow_steps
- opportunites (pipeline)
- factures, facture_lignes
- taches
- activites (journal)

3. Routes API (point par point)
-------------------------------
- `POST /api/auth/login` : connexion (email + mot de passe)
- `POST /api/auth/logout` : destruction de session
- `GET /api/auth/me` : info utilisateur connecté
- `GET/POST/PUT/DELETE /api/contacts` : gestion contacts
- `GET/POST/PUT/DELETE /api/opportunites` : pipeline
- `GET/POST/PUT/DELETE /api/factures` : documents (ajout: `/api/factures/:id/pdf` pour export PDF)
- `GET/POST/PUT/DELETE /api/taches` : tâches
- `GET /api/stats/*` : endpoints analytiques (dashboard, pipeline-par-etape, ca-mensuel, taches-par-statut, etc.)
- `GET/POST /api/users` : gestion utilisateurs (admin)
- `GET/POST /api/services`, `workflows`, `departments`, `export` endpoints

4. Fonctionnalités utilisateur existantes
----------------------------------------
- Authentification par session
- Dashboard avec stats (via Chart.js côté front)
- CRUD contacts, opportunités, factures/devis, tâches, workflows
- Génération PDF côté serveur pour factures/devis (nouvelle feature ajoutée)
- Seed automatique pour remplir de données test

5. Sécurité et points d'attention
---------------------------------
- `SESSION_SECRET` par défaut présent dans `.env.example` et `.env` — doit être changé
- `express-session` utilise MemoryStore : non adapté à la production (utiliser Redis ou autre store)
- Absence de `helmet`, `csurf`, `rate-limit` — ajouter pour production
- Vérifier permissions d'accès aux exports (déjà protégés par `requireAuth`, mais possibilité d'ajouter `requireRole` pour restreindre downloads)
- Passwords stockés hashés (bcrypt)
- .gitignore exclut `.env` et `data/*.db` (OK)

- Services et packs (catalogue, affectation aux contacts).

- Endpoints REST dans `server/routes/` (auth, contacts, opportunites, factures, taches, stats, users, services, workflows, departments, export).

-------------------------

- Initialisation & seed : `server/db/init.js` et `server/db/seed.js` créent tables et insèrent données de test (100 contacts, opportunités, factures, etc.).

14) "Prévision du chiffre d'affaires et du pipeline par séries temporelles"
   - Périmètre : utiliser les données historiques de facturation et d'opportunités pour modéliser le chiffre d'affaires futur et les tendances du pipeline.
   - Objectifs : construire des modèles de prévision (ARIMA/Prophet/simple expo smoothing), analyser les performances et proposer des prévisions mensuelles/quarterly.
   - Livrables : pipeline de préparation des séries temporelles, modèles de prévision, rapports de performance, visualisation de la trajectoire CA.
   - Pourquoi : ce thème est très pertinent pour un data analyst car il relie l'analyse des données à des prévisions business à forte valeur pour les décideurs.
